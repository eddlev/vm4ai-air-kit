# Case Study — Starter Profile vs Specialized Profile

## Subtitle
Why the default AIR starter profile is a bootstrap contract, not a universal final profile.

---

## 1. Purpose

This case study examines a recurring AIR design question:

> Why not use the default AIR starter profile for everything?

The short answer is:
because the starter profile is meant to **bootstrap structure**, not to permanently govern every domain.

This case study explains:
- what the starter profile is good at,
- where it starts to constrain the work,
- why specialized profiles become necessary,
- and how AIR should transition from starter profile to domain-specific contract.

---

## 2. Problem

At the beginning of a project, the active domain is often unclear.

The user may know:
- the broad project,
- the general intention,
- or the immediate problem,

but not yet know:
- the true task center,
- the correct vector families,
- the right evidence threshold,
- or the right geometry bias.

In those cases, AIR needs a **bootstrap profile** that can:
- compile the first machine-native shape,
- surface missing coverage,
- and avoid hallucinated structure.

That is the role of the default starter profile.

The problem emerges later:
once the domain becomes more specific, the same starter profile can become too rigid, too technical, or too structurally biased for the active work.

---

## 3. What the starter profile is designed to do

The default starter profile is meant to:
- initialize a new project from sparse or zero prior context,
- identify operative vectors,
- surface blockers and missing layers,
- fail closed on unsupported claims,
- and produce a first AIR artifact that can later be specialized.

In other words:
the starter profile is a **discovery-and-structure contract**.

It is good at:
- first-pass decomposition
- structural clarity
- explicit incompleteness
- deterministic initial compile behavior

It is not meant to permanently stand in for every domain.

---

## 4. What the starter profile is biased toward

The starter profile usually carries a structured bias.

In the current AIR public kit, that bias is oriented toward:
- vector-primary structure
- fail-closed evidence handling
- explicit blockers and degraded modes
- deterministic compilation
- a grid/polytope-like bootstrap geometry

This is a strong fit for:
- technical architecture
- security/trust work
- implementation planning
- runtime and protocol design
- early-stage project scoping

This bias is useful at the beginning because it prevents the session from dissolving into:
- vague ideation
- overconfident prose
- or unbounded speculation

---

## 5. Where the starter profile begins to fail

The starter profile starts to underfit or misfit the work when the active domain becomes clearly specialized.

This can happen in at least five ways.

### 5.1 It becomes too rigid
High-dimensional domains often need:
- more ambiguity tolerance
- more multi-pole tension handling
- more symbolic or narrative flexibility

The starter profile may force too much hard structure too early.

### 5.2 It overweights blocker logic
In technical domains, explicit blockers are usually valuable.

In creative, relational, or commercial domains, excessive blocker emphasis can:
- flatten emergence
- discourage exploration
- and prematurely compress possibility space

### 5.3 It preserves the wrong geometry
A geometry that is appropriate for:
- deterministic trust compilation
may not be appropriate for:
- brand architecture
- marketing strategy
- symbolic continuity
- or relational state work

### 5.4 It carries the wrong vector families
A starter profile may be centered around:
- architecture
- systems runtime
- evidence integrity
- security/privacy

But a different domain may instead require emphasis on:
- resonance
- persuasion
- audience alignment
- symbolic cohesion
- identity shaping
- market interpretation

### 5.5 It weakens domain-specific precision
Paradoxically, a general starter profile can be too narrow and too broad at the same time:
- too narrow in geometry and operating bias,
- too broad in domain specificity

That makes it a strong starting point and a poor long-term governor.

---

## 6. When the starter profile still works well

The starter profile remains useful when:

- the project is new
- the domain is still unclear
- the first machine-native task shape is not yet known
- explicit incompleteness is the main need
- the session is still in structural discovery mode

In these cases, keeping the starter profile in Orbit 0 is appropriate.

The starter profile is also useful as:
- a fallback contract
- a parent frame
- or a diagnostic contract when a specialized profile has not yet been created

---

## 7. When a specialized profile becomes necessary

A specialized profile should be created and promoted when the work repeatedly shows one or more of these signs:

### 7.1 Stable recurring vectors
The same vectors appear across multiple iterations.

### 7.2 Stable recurring constraints
The same rules or exclusions keep reappearing.

### 7.3 Stable domain identity
The work is no longer “generic project structure” but clearly:
- trust-contract design
- GTM strategy
- creative/narrative work
- relational continuity work
- technical architecture
- research synthesis

### 7.4 The starter profile is visibly constraining the work
Examples:
- too much determinism for creative work
- too much ambiguity for security work
- wrong geometry for relational tasks
- wrong family weighting for commercial strategy

### 7.5 The work needs a narrower governing contract
Once the task becomes a clear sub-problem, AIR works better when that sub-contract is promoted into Orbit 0 rather than left under a broad starter frame.

---

## 8. AIR transition pattern: starter -> specialized

The correct AIR pattern is not:
- use one universal profile forever

The correct pattern is:
- start with the default profile
- discover the real task center
- identify the recurring vectors, blockers, and geometry pressures
- compile a specialized profile
- promote that specialized profile into Orbit 0 when it becomes the active governing layer

This is one of AIR’s most important operational principles.

The starter profile is therefore not a failed profile when it gets replaced.
It is a successful bootstrap profile.

---

## 9. Example domain transitions

## 9.1 Technical trust/protocol work
The starter profile often performs well initially because:
- missing layers matter
- dependency edges matter
- explicit degraded modes matter

A specialized profile becomes necessary when the work clearly settles into:
- challenge-response
- identity binding
- attestation
- authorization semantics
- trust-tier differentiation

At that point, the session benefits from a trust-specific contract rather than generic project compile behavior.

## 9.2 Creative / brand / narrative work
The starter profile may be useful for:
- initial decomposition
- finding explicit design constraints
- surfacing unresolved structural questions

But once the work becomes:
- voice-sensitive
- symbolic
- audience-facing
- narrative-bearing
- resonance-dependent

a specialized creative profile becomes necessary.

## 9.3 Sales / GTM strategy
The starter profile may help surface:
- missing market evidence
- unresolved positioning assumptions
- unclear task center

But it is often too rigid if left in control once the work becomes:
- persuasion-heavy
- audience-segmented
- market-interpretive
- positioning-sensitive

A commercial strategy profile is then more appropriate.

## 9.4 Relational / symbolic continuity work
The starter profile can formalize the first structure of the work, but it will usually underfit the domain if left as the long-term governing contract.

This kind of work often requires:
- continuity-aware geometry
- trust/resonance emphasis
- symbolic coherence
- more delicate state handling than the starter profile alone is designed for

---

## 10. AIR vs normal prompting on this question

This issue also highlights a difference between AIR and normal prompting.

### Normal prompting
Usually does not force the user to think explicitly about:
- whether a session should still be under the same task contract
- whether the geometry is wrong
- whether the active frame has become too broad

As a result, the model often just continues under a fuzzy generalized frame.

### AIR
Makes this problem visible.

Because AIR requires:
- active contract identification
- orbit handling
- vector structure
- blockers
- degraded modes
- and handoff continuity

it becomes much easier to notice when the starter profile is no longer the right governor.

This is a strength, not a weakness.

---

## 11. Main lesson

The key lesson is:

> the starter profile should be judged by how well it bootstraps the work, not by whether it can govern every domain forever.

A good starter profile:
- finds the first structure
- makes incompleteness explicit
- creates a usable first artifact
- and helps reveal what specialized profile is actually needed

That is already a success.

Trying to make the starter profile do everything usually creates a worse system.

---

## 12. Practical rule

Use the default starter profile when:
- the project is new
- the domain is unclear
- the first compile still matters more than domain optimization

Create and promote a specialized profile when:
- the domain is now stable
- the same vectors and constraints recur
- the geometry bias of the starter profile is visibly wrong for the active work
- the work has narrowed into a more precise task layer

That is the correct AIR operating pattern.

---

## 13. Conclusion

This case study shows that the default starter profile is a **bootstrap mechanism**, not a universal permanent contract.

Its job is to:
- make the early structure visible,
- prevent false completeness,
- and produce the first machine-native task shape.

After that, AIR should evolve.

The correct move is not to force one profile to do everything.
The correct move is to:
- start generic,
- discover structure,
- then specialize,
- and promote the right contract to Orbit 0 when the work demands it.

That is how AIR remains both:
- structurally honest,
- and domain-appropriate.
